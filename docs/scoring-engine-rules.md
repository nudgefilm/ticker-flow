# 스코어링 엔진 (어드민 3개 섹션) 작업 규칙

> 이 문서는 평소 작업 시 참고할 필요는 없다. `/admin` 홈의 스코어링 엔진(티커플로우
> 스크리너 / Top 30 Ticker Overlay / TOP30 Entry·Outcome 추적) 관련 작업을 할 때만
> 열어서 확인한다. CLAUDE.md와는 독립적으로 관리하며, CLAUDE.md에서 링크하지 않는다.
> 작성: 2026-07-11

배경: 스코어링 엔진 재작성(세션 56) 과정에서 기존 반영 항목(analyst_ratings)이 사용자
확인 없이 조용히 누락됐고, 이후 여러 세션 동안 아무도 인지하지 못한 채 방치됨. 동시에
계산 로직은 다음 세대(13개 팩터 체계)로 넘어갔는데 화면 표기 문구는 갱신되지 않아,
조사할 때마다 화면과 실제 코드가 어긋나 혼선을 유발함.

---

## 1. 대상 범위

아래 3개 섹션은 모두 **스코어링 엔진**(`src/lib/collect/scoring.ts`의 `computeScores()`)이
계산한 단일 결과를 공유해서 보여주는 화면이다. 셋 중 하나라도 관련된 요청은 이 문서의
규칙을 적용한다.

- 티커플로우 스크리너 (TickerFlow Screener) — `AdminWatchSection()`, `src/app/admin/page.tsx`
- Top 30 Ticker Overlay — `Top30OverlaySection()` → `getTop30OverlayData()`, `src/app/admin/top30-overlay-data.ts`
- TOP30 Entry/Outcome 추적 — `OutcomesSection`, `src/app/admin/outcomes-section.tsx`

## 2. 데이터 흐름 구조 (반드시 숙지 후 작업)

```
[매일 22:35 KST] /api/collect/top30 크론
        ↓
computeScores() 실행 (src/lib/collect/scoring.ts)
  — 13개 확정 팩터(CLAUDE.md 18항, src/lib/scoring/weights.ts의 SCREENER_WEIGHTS) 기준 계산
        ↓
결과 저장: top30_daily / top30_entries / top30_outcome_results
        ↓
3개 섹션은 전부 "읽기 전용" — 위 스냅샷 테이블을 각자 다른 각도로 표시만 함
```

**핵심 원칙**: 스코어링 엔진 로직 수정은 `computeScores()`(및 `weights.ts`) **한 곳만**
고치면 다음 크론 실행 시 3개 섹션 전체에 자동 반영된다. 3개 섹션의 렌더링 파일 자체는
스코어링 계산과 무관하다.

**동시에 이는 리스크이기도 하다**: `computeScores()` 한 곳에서 뭔가 조용히 빠지거나
바뀌면, 3개 섹션 모두 티 안 나게 동시에 영향을 받는다. 재작성 시 특히 주의.

## 3. 작업 규칙

### 3-1. 기존 팩터·데이터 소스 제거 시 사전 확인 필수
- 스코어링 엔진을 리팩토링·재작성할 때, 기존에 반영되어 있던 팩터·데이터 소스를
  제거해야 하는 경우 **임의로 제거하지 않는다.**
- 제거하기 전 반드시 "OO 팩터/데이터를 이번 개편에서 빼려고 하는데 괜찮은가"를 먼저
  확인받는다.
- 완료 보고에 "제거했다"는 사후 기록만 남기는 것은 사전 확인을 대체하지 못한다.
- "전면 개편/재작성"류 작업은 시작 전 기존 로직에서 유지·이관해야 할 항목 목록을 먼저
  정리해 확인받은 뒤 진행한다.

### 3-2. 계산 로직 변경 시 화면 표기 문구 동시 갱신 필수
- 가중치·팩터 구성이 바뀌면, 이를 설명하는 화면 텍스트(예: 어드민 페이지의 "OO% + OO%"
  식 라벨)도 **같은 작업 안에서 반드시 함께 갱신**한다.
- 계산 로직만 바꾸고 화면 문구를 그대로 두는 것은 금지 — 실제 로직 세대와 화면 설명
  세대가 어긋나면 이후 조사·판단이 왜곡된다.
- 작업 완료 보고 시 "화면 표기 문구 갱신 여부"를 별도 항목으로 명시한다.

### 3-3. CLAUDE.md 18항 확정 13개 팩터 구조 변경은 별도 승인 필요
- 현재 13개 확정 팩터(활성 8개, 합계 78%) 구조를 벗어나 새 팩터를 추가/삭제하거나
  가중치 배분을 바꾸는 것은 "단순 수정"이 아니라 **가중치 체계 재설계**에 해당한다.
- CLAUDE.md 10항 규칙에 따라 이런 변경은 진행 전 Opus 전환을 권장 고지한다.
- 13개 확정 리스트 자체를 개정(14번째 팩터 추가 등)할지 여부는 정책 결정이므로, 코드
  작업 이전에 이 결정부터 사용자에게 확인받는다.

### 3-4. 완료 보고 무검증 전달 금지 (이 영역 한정 강화 적용)
- 이 3개 섹션과 관련된 Claude Code의 "완료" 보고는 특히 더 엄격하게 검증한다.
- 코드 수정 완료 보고만으로 끝내지 않고, **실제로 3개 섹션 화면에 값이 바뀌어
  나타나는지**까지 확인 후 사용자에게 전달한다.
- 조사 요청(누락 확인, 구조 확인 등)에 대한 응답을 받으면, 표면적 답변으로 마무리하지
  않고 CLAUDE.md 등 문서 기록과 실제 코드가 일치하는지 대조하는 절차를 기본으로
  포함한다.

### 3-5. 과거 발생 문제 이력 (참고)
- **analyst_ratings 누락**: 세션 56(스코어링 엔진 전면 개편)에서 애널리스트 투자의견
  분포(`analyst_ratings` 테이블) 반영 로직이 사용자 확인 없이 통째로 빠짐. 이후 여러
  세션 동안 발견되지 않고 방치됨. 수집 크론(`/api/collect/analyst`, 매일 09:25 KST)은
  계속 돌아가고 있었으나 소비처가 없는 상태로 방치됨.
- **화면 문구 세대 불일치**: 계산 로직은 5영역 가중치 체계 → 13개 팩터 체계(CLAUDE.md
  18항)로 이미 넘어갔으나, `admin/page.tsx`의 화면 라벨("스마트머니 45% + 실적품질
  30% + ...")은 옛 5영역 체계 문구 그대로 방치됨.

---

**원칙 요약**: 스코어링 엔진은 3개 섹션에 단일 계산 결과를 공급하는 구조이므로 한 곳
수정의 파급력이 크다. 기존 항목 제거는 반드시 사전 확인, 로직 변경은 화면 문구와 함께
갱신, 완료 보고는 화면 기준으로 재검증. 13개 확정 팩터 구조 자체를 건드리는 변경은
별도 정책 승인 후 Opus 전환.
