# 1. 프로젝트 개요

* 서비스명: TickerFlow
* 도메인: tickerflow.net
* 운영: 언폴드랩 (UNFOLD LAB)
* 운영 형태: 1인 개발 SaaS
* 대상: 미국 나스닥 주식에 투자하는 국내 개인 투자자(서학개미)

## 서비스 정의

TickerFlow는 투자 자문 서비스가 아니다.

TickerFlow는 공개된 정보를 수집·정리·분류·시각화하여 사용자가 미국 기업의 중요한 변화를 빠르게 파악할 수 있도록 돕는 모니터링 대시보드이다.

사용자는 투자 아이디어를 얻기 위해 방문하는 것이 아니라,

* 어떤 기업에 변화가 있었는지
* 어떤 공시가 등록되었는지
* 관심 종목에 새로운 이벤트가 발생했는지

를 확인하기 위해 방문한다.

## 슬로건

미국 기업의 중요한 변화, 놓치지 마세요

# 10. Claude Code 작업 규칙

1. 항상 CLAUDE.md를 먼저 읽는다.
2. 실제 파일 구조 확인 후 작업한다.
3. 실제 DB 구조 확인 후 구현한다.
4. 환경변수 중복 생성 금지.
5. 테스트 데이터를 프로덕션에 삽입하지 않는다.
6. 원인 없는 수정 금지.
7. 작업 완료 전 임의 종료 금지.
8. 서비스 전체에서 "AI" 단어 사용 금지.
9. 모든 페이지 하단에 면책 문구 포함.
10. 랜딩은 HSL 토큰, 대시보드는 HEX 토큰 사용.
11. 투자 권유, 투자 추천, 투자 자문으로 해석될 수 있는 UI·카피·점수 체계를 생성하지 않는다.
12. "정보 제공"과 "변화 모니터링" 원칙을 최우선으로 유지한다.
13. import 수정 시 기존 import 목록을 먼저 확인 후 추가만 한다. 삭제 금지.
14. str_replace로 import 블록 수정 시 전체 블록을 교체하지 않는다. 추가할 항목만 별도 라인으로 삽입한다.
15. `pnpm gen:types` 실행 후 커밋 전 반드시 `src/types/supabase.ts` 첫 줄이 `export type Json =`으로 시작하는지 확인한다. 아니면 커밋 금지 — `git checkout HEAD -- src/types/supabase.ts`로 복원.
16. Supabase 타입에 없는 테이블(예: collect_runs)은 `(adminClient as any).from("테이블명")` 패턴으로 쿼리한다.
17. 새 테이블 생성 시 반드시 아래 4가지를 함께 작성한다:
    ① `supabase/schema.sql`에 테이블 정의 추가
    ② `authenticated` role SELECT 권한 GRANT
    ③ `service_role` GRANT (관리자 클라이언트 사용 테이블)
    ④ RLS 정책 추가 ("authenticated can select ...")
    위 4가지 중 하나라도 빠지면 커밋 금지.

# 13. 디자인 시스템 운영 원칙

대상

규칙

* globals.css HSL 토큰 사용
* bg-background
* text-foreground

* 직접 HEX 사용 금지

## 앱 대시보드

대상

* /dashboard 이하 모든 페이지

규칙

* 직접 HEX 사용
* HSL 토큰 사용 금지

| 용도   | HEX       |
| ---- | --------- |
| 배경   | #0a0a0a   |
| 사이드바 | #0f0f0f   |
| 카드   | #1a1a1a   |
| hover/강조면 | #212121 |

랜딩 페이지 `--card` 토큰(`hsl(0 0% 10%)`)은 위 대시보드 카드 기준(#1a1a1a)과 동일한 밝기로 통일한다.

## 앱 대시보드 텍스트 색상 기준

랜딩 페이지의 `muted-foreground`(`hsl(0 0% 65%)`)는 HEX `#a6a6a6`과 동일하다.
대시보드에서 보조 텍스트는 반드시 아래 기준을 따른다.

| 용도             | HEX       |
| -------------- | --------- |
| 주 텍스트          | #ffffff   |
| 보조 텍스트 (강조)    | #cccccc   |
| 보조 텍스트 (기본)    | #a6a6a6   |

`#666666` 이하의 어두운 회색은 대시보드 텍스트에 사용하지 않는다.

# 14. 번역 프롬프트 운영 원칙

번역 프롬프트는 `src/lib/collect/summarize.ts`에서 관리한다.

## 기본 원칙

* 공시·뉴스에 명시된 사실만 요약한다.
* 분석, 해설, 의견, 전망은 추가하지 않는다.
* "~했습니다", "~발표했습니다" 형태의 사실 서술체로만 작성한다.
* 투자 판단과 관련된 표현은 중립적으로 서술한다.
* plain text로만 응답한다. 마크다운 기호(#, **, - 등) 사용 금지.

## 분량 기준

| 대상 | 분량       |
| -- | -------- |
| 공시 | 3~5문장, 200자 내외 |
| 뉴스 | 2~3문장, 100자 내외 |

## 규제 대응

외부 기관(금융감독원, 공정거래위원회 등)으로부터 서비스 내 텍스트에 대한 질의가 있을 경우,
이 섹션과 `src/lib/collect/summarize.ts`의 실제 프롬프트 코드를 근거로 제시한다.

* 요약 생성 모델: Claude Haiku (`claude-haiku-4-5-20251001`)
* 프롬프트 원칙: 팩트 서술만, 투자 권유 표현 배제
* 생성 결과는 원문 공시·뉴스 링크와 함께 표시되어 사용자가 원문을 직접 확인 가능

## 콘텐츠 성격별 면책 문구 구분 원칙

대시보드 화면용 표준 면책 문구("본 서비스는 공개된 정보를 기반으로...")는
TickerFlow 서비스 화면(UI)을 지칭하는 표현이므로, 서비스 화면 밖으로
나가는 콘텐츠(블로그 포스팅, 유튜브 쇼츠 등)에는 그대로 재사용하지 않는다.

* 서비스 화면(대시보드, 마이페이지 등): `DashboardDisclaimer` 표준 3줄
  ("본 서비스는...", "특정 종목에 대한...", "투자 판단과 결과에 대한...")
* 서비스 화면 밖 콘텐츠(블로그, 유튜브 쇼츠 등): "TickerFlow Note" 문구

  ```
  TickerFlow Note
  이 글은 미국 기업의 공시와 시장 데이터를 기반으로 주요 변화를 정리한 콘텐츠입니다.
  TickerFlow는 공개된 데이터를 바탕으로 기업 활동과 시장 흐름을 모니터링하며, 특정 종목에 대한 투자 권유나 투자 자문을 제공하지 않습니다.
  투자 판단은 다양한 정보를 함께 검토한 뒤 본인의 책임하에 결정하시기 바랍니다.
  ```

* 두 문구 모두 모델이 생성하지 않고 코드에서 고정 상수로 삽입한다
  (블로그 초안: `src/lib/collect/blog-draft.ts`의 `DISCLAIMER` 상수).
* 향후 새 콘텐츠 채널을 추가할 때도 "서비스 화면인가, 화면 밖 콘텐츠인가"를
  기준으로 이 중 맞는 문구를 사용한다.

# 16. Supabase 타입 운영 원칙

## 자동 생성 타입

`src/types/supabase.ts`는 Supabase CLI로 자동 생성한다. 수동으로 편집하지 않는다.

```bash
pnpm gen:types   # 또는 npm run gen:types
```

스키마 변경 후 반드시 재생성한다.

## 조인 쿼리 타입 패턴

Supabase `.select()` 조인 쿼리 결과는 PostgREST 내부 타입과 불일치가 발생할 수 있다.
이 경우 항상 아래 패턴을 사용한다.

```typescript
// 직접 선언한 타입 (nullable 필수)
type MyRow = {
  ticker: string;
  tickers: { name_kr: string | null; name_en: string | null } | null;
};

// 캐스트는 항상 as unknown as MyType[]
const rows = (data ?? []) as unknown as MyRow[];
```

규칙

* 조인된 관계 필드는 항상 `| null`을 붙인다.
* `as MyType[]` 직접 캐스트는 사용하지 않는다. `as unknown as MyType[]`만 사용한다.
* 생성된 `Database` 타입은 세 클라이언트(`server.ts`, `client.ts`, `admin.ts`)에 모두 주입한다.

# 18. 티커플로우 스크리너 가중치 구조 (내부 전용, 확정)

티커플로우 스크리너는 어드민(`/admin`) 전용 내부 도구이며, 화면에서 빨간 테두리

사용자에게 노출되는 화면·API에는 이 점수 체계를 절대 포함하지 않는다
(11항 최상위 판단 기준, "종합 투자 점수" 금지 원칙 참고).

## 최종 목표 구조 (13개 항목, 100% 확정)

가중치 정의는 `src/lib/scoring/weights.ts`의 `SCREENER_WEIGHTS`에서 관리한다.

| 기관수급(13F) | 15% | 활성 |
| 내부자거래 | 10% | 활성 |
| 목표주가변화 | 9% | 활성 |
| 기업이벤트(공시) | 12% | 활성 |
| 뉴스 | 5% | 활성 |
| 공매도변화 | 5% | 활성 |
| 가격모멘텀 | 4% | 활성 |

활성 항목끼리만 정규화(활성 가중합산 점수 ÷ 활성 비중 합계 × 100)하여 항상
100점 만점 체계로 산출한다.

## 원칙

* 이 13개 항목·최종 비중은 확정 구조이며, 향후 데이터가 추가되어도 비중 재분배
  없이 해당 항목의 `active` 플래그만 `true`로 전환한다. 다른 항목의 비중은
  그대로 유지된다.
* 신규 팩터를 추가하고 싶어도 이 13개 리스트에 항목을 늘리지 않는다.
* 4단계(Relative Strength, 목표가 괴리율 등 신규 팩터 후보)는 최소 수개월간의
  실운영 후 `factor_log` 기반 성과 검증을 거쳐 별도로 설계·검토한다.
* 5단계(예측 경로 시각화)는 4단계 신규 팩터 검증까지 완료된 근거 기반 모델이
  확립된 이후, Top 30 Ticker Overlay 차트에 향후 1년 예측 가격 경로를 점선으로
  반영하는 것을 목표로 한다. 검증 안 된 상태에서는 절대 예측선을 표시하지 않는다.
* 팩터별 raw score는 `top30_daily.factor_log`(jsonb)에 내부 로그로만 저장하며,
  사용자 노출 API/화면에는 절대 포함하지 않는다. 비활성 항목 또는 데이터
  미존재 시 값은 `null`("계산 안 함")이며, 활성 항목의 계산 결과 0과 구분한다.

## 2.5단계 — 검증 인프라 (배점 설계 아님)

2.5단계는 배점 설계가 아니라 검증 인프라 구축이다. TOP30 선정 시점의
`factor_log`를 스냅샷으로 남기고, 이후 실제 가격 성과(30/60/90일)를 추적해
`top30_entries`/`top30_outcome_results`에 최소 2~3개월 데이터가 축적된 후
팩터별 예측력을 실측 데이터로 분석하여 4단계(배점 설계)를 진행한다.

* `SCORING_MODEL_VERSION`(`src/lib/scoring/version.ts`)은 `weights.ts`의
  `SCREENER_WEIGHTS` 변경 시 반드시 함께 갱신하며, `top30_daily`·
  `top30_entries` 양쪽에 저장되어 어느 시점의 TOP30이 어떤 모델 버전으로
  계산됐는지 항상 추적 가능해야 한다. 현재는 `version.ts`를 단일 진실
  (Single Source of Truth)로 사용하며, 향후 운영 중 동적 버전 변경이
  필요해질 경우에만 DB 설정 테이블로 이전한다.
* `TRACKED_DAYS`(`src/lib/outcomes/config.ts`)로 성과 추적 기간을 중앙
  관리하며, 임의 기간(180일, 365일 등) 확장이 스키마 변경 없이 가능하다.
* `top30_outcome_results`의 모든 계산은 `entry_id`를 유일한 기준으로 하며,
  `ticker`/`selected_date`는 조회용 비정규화 컬럼일 뿐이다.
* `top30_entries`는 생성 이후 수정하지 않는 불변(immutable) 스냅샷이다.
  이후 모델 버전 변경이나 factor_log 계산 로직 수정이 발생해도 과거 Entry는
  그대로 유지된다.
  * **좁은 예외 — entry_price 백필**: TOP30 선정이 장중(13:35 UTC)에
    실행되어 선정 시점에는 당일 종가가 존재하지 않을 수 있다. 이 경우
    `entry_price`는 null로 생성되고, `top30-outcomes.ts`가 매일 실행될 때
    `entry_price IS NULL`인 행만 최초 1회 채운다. 이미 값이 채워진 행은
    절대 다시 덮어쓰지 않는다 — "확정된 값의 수정"이 아니라 "아직 확정
    안 된 null을 최초로 채우는 것"이므로 불변 원칙 위반이 아니다.
* `top30_entries` 및 `top30_outcome_results`는 분석용 원본 데이터셋으로
  간주한다. 운영 중 DELETE를 수행하지 않으며, 데이터 수정이 필요한 경우에도
  기존 행을 변경하지 않고 새로운 Entry/Event를 생성하는 방향을 우선한다
  (단, 명백한 수집 오류에 대한 관리자 수동 보정은 예외).
* 이 테이블들과 관련 화면은 어드민 전용 규제 예외 구간에 속하며, 사용자
  노출 API·화면에 절대 포함하지 않는다.
* `weights.ts`의 `SCREENER_WEIGHTS` `active` 플래그는 2.5단계에서 변경하지
  않는다.
